System map

Hooks capture the work. SQLite keeps the truth.

Eagle Mem is intentionally boring in the best way: agent hooks, a local SQLite database, FTS5 search, and small shell scripts. The power comes from where it sits in the loop, not from a hosted memory service.

Agent Event

Claude Code or Codex starts, receives a prompt, uses a tool, or stops a turn.

->

Eagle Hook

Hooks inject recall, record observations, block risky commands, or summarize work.

->

Memory Core

SQLite tables and FTS5 indexes store project memory with source attribution.

Runtime layers

The moving parts are small.

Each layer owns a clear responsibility. That keeps Eagle Mem easy to install, inspect, uninstall, and repair when a user's local environment is unusual.

Layer 1

Hook lifecycle

SessionStart, UserPromptSubmit, PreToolUse, PostToolUse, Stop, and SessionEnd provide the capture and injection points.

Layer 2

Project memory

Summaries, decisions, gotchas, memories, plans, tasks, and project overview all land in one project namespace.

Layer 3

Feature safety

Changed feature files create fingerprinted pending checks. Release commands stay blocked until the current diff is verified or waived.

Layer 4

Worker lanes

Broad work can split into git worktrees with owned lanes, validation commands, worker logs, and durable handoff state.

Data model

One database, multiple memory surfaces.

The same database powers hook recall, CLI search, mirrored agent memories, feature verification, and orchestration status.

sessions Tracks active and completed Claude Code or Codex sessions.
summaries Stores rich Stop-hook summaries with decisions, gotchas, files, and risks.
agent_memories, agent_plans, agent_tasks Mirror Claude Code and Codex memory surfaces with `origin_agent`.
features, feature_files, pending_feature_verifications Connect code changes to release verification and smoke-test state.
orchestrations, orchestration_lanes Persist worker ownership, status, worktree paths, logs, notes, and handoffs.
code_chunks FTS5-indexed source chunks for lightweight code recall.
Shell output

RTK is a guardrail, not a dependency.

When `rtk` is installed, Eagle Mem can rewrite noisy commands through it. In enforce mode, known raw-output commands are blocked if RTK is missing. 

eagle-mem config set token_guard.rtk auto
eagle-mem config set token_guard.rtk enforce
eagle-mem config set token_guard.raw_bash block
Token Guard git diff -> rtk git diff git show -> rtk git show grep / rg / find -> blocked in enforce mode Goal Keep huge shell output from entering the model context raw. Escape hatch Temporary unlocks exist for intentional one-off raw output.
Orchestration

Opposite-agent work by default.

The coordinator can be Claude Code or Codex. By default Eagle Mem routes workers to the other agent family, records model and effort, and runs lanes in isolated git worktrees so handoff is reviewable.

Codex coordinator

Defaults to Claude Code workers with `claude-opus-4-7` and `xhigh` effort when the local CLI is available.

Claude Code coordinator

Defaults to Codex workers with `gpt-5.5` and `xhigh` effort when the local CLI is available.
